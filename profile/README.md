## 🤝 Contribution Guidelines

We believe that great software is built through collaboration. To maintain a high standard of code quality and a maintainable history, we ask all contributors to adhere to the following workflow:

### 1. The Rebase Workflow
To keep our project history clean, linear, and easy to audit, we strictly follow a **rebase-only** merge policy. 
*   Avoid merge commits that clutter the tree. 
*   If you are new to `git rebase`, don't worry! Open your Pull Request, and our maintainers will gladly guide you through the process or help you squash and rebase your branch before the final merge.

### 2. Meaningful Commits
Your commit history is a story for future developers. Please make it readable:
*   **Atomic Changes:** Keep commits focused. One commit should ideally address one specific fix or feature.
*   **Descriptive Messages:** Avoid vague titles like "fixed bug" or "update". Explain *what* changed and *why*.
*   **Conventional Commits:** We prefer (but don't strictly enforce) the use of prefixes:
    *   `feat:` for new features.
    *   `fix:` for bug fixes.
    *   `refactor:` for code changes that neither fix a bug nor add a feature.
    *   `chore:` for maintenance tasks or documentation updates.

### 3. Review Process
All code must be reviewed by at least one maintainer. We value constructive feedback and may ask for adjustments to ensure consistency with the rest of the codebase.
